## Mensagem 1

**Usuário:**

> A integração entre o backend e o frontend não estão funcionando, pode corrigir?

---

**Assistente:**

Explorei a estrutura do projeto e identifiquei os seguintes problemas:

### Problemas encontrados

1. **`chat/templates/chat/chat.html`** usava IDs e estrutura HTML completamente diferentes dos que os módulos JS esperam — e não carregava nenhum arquivo estático.
   - Template usava: `#chat-window`, `#message-input`, `#chat-form` com JS inline
   - Módulos JS esperam: `#chat-messages`, `#chat-input`, `#chat-send`, `#chat-error`, `#chat-typing`, `#chat-new`, `#chat-theme-toggle`

2. **`chatApp.js`** tinha bloco `try/finally` sem `catch` — erros do backend eram descartados silenciosamente sem exibir nada ao usuário.

3. **`chatApi.js`** lançava mensagem genérica em erros HTTP, sem ler o corpo da resposta do servidor.

### Correções aplicadas

**`chat/templates/chat/chat.html`** — Reescrito para:
- Usar a estrutura correta com os IDs que os módulos JS esperam
- Carregar `chat.css` e `chatApp.js` via `{% load static %}` / `{% static %}`
- Manter `{% csrf_token %}` no formulário

```html
{% load static %}
<!DOCTYPE html>
<html lang="pt-BR">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>Ajuda Tech — Chat</title>
    <link rel="stylesheet" href="{% static 'chat/css/chat.css' %}" />
  </head>
  <body>
    <main id="chat-app">
      <header class="chat-header">
        <div class="chat-header-row">
          <div>
            <h1>Ajuda Tech</h1>
            <p class="chat-subtitle">Encontre o computador ideal para você</p>
          </div>
          <button id="chat-theme-toggle" type="button" class="chat-theme-toggle"
            aria-label="Ativar modo escuro" title="Ativar modo escuro"></button>
        </div>
      </header>
      <div id="chat-messages" class="chat-messages" aria-live="polite"></div>
      <p id="chat-typing" class="chat-typing" hidden>Digitando...</p>
      <p id="chat-error" class="chat-error" hidden role="alert"></p>
      <form class="chat-form">
        {% csrf_token %}
        <label class="visually-hidden" for="chat-input">Sua mensagem</label>
        <textarea id="chat-input" rows="3"
          placeholder="Descreva o que você precisa fazer no computador..."></textarea>
        <div class="chat-actions">
          <button id="chat-send" type="button">Enviar</button>
          <button id="chat-new" type="button" class="chat-new">Nova conversa</button>
        </div>
      </form>
    </main>
    <script type="module" src="{% static 'chat/js/chatApp.js' %}"></script>
  </body>
</html>
```

**`chat/static/chat/js/chatApp.js`** — Adicionado `catch` no `handleSend`:

```javascript
try {
  const response = USE_MOCK
    ? await postChatMock(userText)
    : await postChat(userText, null);
  state = appendMessage(state, { role: 'bot', text: response.reply });
  refresh();
} catch (err) {
  showError(errorEl, err.message || 'Não foi possível obter resposta. Tente novamente.');
} finally {
  setTypingVisible(typingEl, false);
  setSendDisabled(sendBtn, false);
  inputEl.focus();
}
```

**`chat/static/chat/js/chatApi.js`** — `postChat` agora parseia o JSON de erro:

```javascript
if (!response.ok) {
  let errorMsg = 'Falha ao enviar mensagem';
  try {
    const errorData = await response.json();
    if (errorData.error) errorMsg = errorData.error;
  } catch (_) {
    // ignora erros de parse — usa mensagem genérica
  }
  throw new Error(errorMsg);
}
```

Todos os 105 testes (backend + frontend) passando após as correções.

## Mensagem 3

**Usuário:**

> Aparentemente funcionou, porém só retorna erro "Não foi possível processar a resposta."
>
> Preciso que verifique e corrija este ponto, e para facilitar a identificação de futuros problemas implemente um rastreamento de logs em arquivos .log

---

**Assistente:**

### Diagnóstico

Testando a API diretamente, foi identificado que o modelo `deepseek/deepseek-v4-flash:free` retornava **HTTP 402** com a mensagem:

```json
{
  "error": {
    "message": "Provider returned error",
    "code": 402,
    "metadata": {
      "raw": "{\"error\":{\"type\":\"insufficient_quota\",\"message\":\"Out of credits. Top up at /dashboard/billing to continue.\"}}"
    }
  }
}
```

O modelo estava roteando por um provider pago ("Crucible") que exigia créditos. O código tratava 402 como `InvalidResponseError` genérico → view retornava a mensagem vaga.

Após testar vários modelos gratuitos da API do OpenRouter, o modelo **`nvidia/nemotron-3-super-120b-a12b:free`** foi confirmado como funcional.

### Correções aplicadas

**`.env`** — modelo atualizado:
```env
LLM_MODEL=nvidia/nemotron-3-super-120b-a12b:free
```

**`ajuda_tech/settings.py`** — default atualizado + logging em arquivo configurado:

```python
import logging.handlers
from pathlib import Path
from decouple import config, Csv

# ...

LLM_MODEL = config("LLM_MODEL", default="nvidia/nemotron-3-super-120b-a12b:free")

# ─── Logging ─────────────────────────────────────────────────────────────────
LOG_LEVEL = config("LOG_LEVEL", default="INFO")
LOG_DIR = BASE_DIR / "logs"
LOG_DIR.mkdir(exist_ok=True)

LOGGING = {
    "version": 1,
    "disable_existing_loggers": False,
    "formatters": {
        "verbose": {
            "format": "{levelname} {asctime} {module} {process:d} {thread:d} {message}",
            "style": "{",
        },
        "simple": {
            "format": "{levelname} {asctime} {module} {message}",
            "style": "{",
        },
    },
    "handlers": {
        "console": {
            "class": "logging.StreamHandler",
            "formatter": "simple",
        },
        "app_file": {
            "class": "logging.handlers.RotatingFileHandler",
            "filename": LOG_DIR / "app.log",
            "maxBytes": 5 * 1024 * 1024,  # 5 MB
            "backupCount": 5,
            "encoding": "utf-8",
            "formatter": "verbose",
        },
        "error_file": {
            "class": "logging.handlers.RotatingFileHandler",
            "filename": LOG_DIR / "errors.log",
            "maxBytes": 5 * 1024 * 1024,  # 5 MB
            "backupCount": 5,
            "encoding": "utf-8",
            "level": "WARNING",
            "formatter": "verbose",
        },
    },
    "root": {
        "handlers": ["console", "app_file", "error_file"],
        "level": LOG_LEVEL,
    },
    "loggers": {
        "django": {
            "handlers": ["console", "app_file", "error_file"],
            "level": "INFO",
            "propagate": False,
        },
        "django.request": {
            "handlers": ["console", "error_file"],
            "level": "WARNING",
            "propagate": False,
        },
        "chat": {
            "handlers": ["console", "app_file", "error_file"],
            "level": LOG_LEVEL,
            "propagate": False,
        },
    },
}
```

**`chat/services.py`** — HTTP 402 agora tratado explicitamente com mensagem real do provider:

```python
if status == 402:
    try:
        detail = response.json().get("error", {}).get("message", "")
    except ValueError:
        detail = response.text[:200]
    raise ServiceUnavailableError(
        f"Créditos insuficientes ou modelo indisponível (HTTP 402): {detail}"
    )
```
